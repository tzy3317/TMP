# Cue-Pro 项目需求文档 (Requirements)

> 本文档定义 Cue-Pro 项目的功能需求与非功能需求，作为开发、测试和验收的依据。
> 版本：v1.0.0 | 最后更新：2026-05-06
> 依赖规范：[spec.md](spec.md)

---

## 1. 项目背景与目标

### 1.1 背景

Cue-Pro 是一个基于 Python 开发的 LLM 应用项目，通过 OpenAI 兼容协议对接各类大语言模型服务，需要提供标准化的环境配置管理能力。

### 1.2 目标

1. 建立一套**规范化的环境变量配置体系**，确保配置清晰、安全、可维护
2. 提供**最小可用配置集**，降低项目启动门槛
3. 预留**扩展接口**，支持未来多模型、多服务商场景

---

## 2. 功能需求 (Functional Requirements)

### 2.1 环境变量配置管理

#### FR-001 必配项强制校验

| 属性 | 说明 |
|------|------|
| **需求编号** | FR-001 |
| **需求描述** | 系统启动时必须校验三个必配环境变量是否存在且非空 |
| **必配项** | `LLM_BASE_URL`、`LLM_MODEL`、`LLM_API_KEY` |
| **校验时机** | 应用启动时 |
| **失败处理** | 抛出明确错误，提示用户检查 `.env` 文件 |
| **优先级** | P0（必须实现） |

**验收标准：**
- [ ] 任一必配项缺失时，应用启动失败并输出友好错误信息
- [ ] 错误信息包含变量名和参考文档链接
- [ ] 校验逻辑位于配置加载模块，不分散在各业务代码中

---

#### FR-002 环境变量加载

| 属性 | 说明 |
|------|------|
| **需求编号** | FR-002 |
| **需求描述** | 支持从 `.env` 文件加载环境变量 |
| **实现方式** | 使用 `python-dotenv` 库 |
| **加载顺序** | 先加载系统环境变量，`.env` 文件中的值可覆盖系统变量 |
| **优先级** | P0（必须实现） |

**验收标准：**
- [ ] 存在 `.env` 文件时，自动加载其中定义的环境变量
- [ ] 不存在 `.env` 文件时，使用系统环境变量（便于容器化部署）
- [ ] 加载过程静默执行，不输出调试信息

---

#### FR-003 配置集中管理

| 属性 | 说明 |
|------|------|
| **需求编号** | FR-003 |
| **需求描述** | 所有配置项通过单一配置模块暴露，禁止直接调用 `os.getenv` 散落于业务代码 |
| **配置模块** | `config.py` |
| **暴露形式** | 字典或 Pydantic 模型 |
| **优先级** | P0（必须实现） |

**验收标准：**
- [ ] 业务代码通过 `from config import LLM_CONFIG` 获取配置
- [ ] 配置模块在导入时自动完成校验和加载
- [ ] 配置值只读，运行时不允许动态修改

---

### 2.2 LLM 调用能力

#### FR-004 OpenAI 兼容协议支持

| 属性 | 说明 |
|------|------|
| **需求编号** | FR-004 |
| **需求描述** | 通过配置的环境变量，能够成功调用任意 OpenAI 兼容协议的 LLM 服务 |
| **协议标准** | OpenAI API v1 |
| **支持操作** | Chat Completions |
| **优先级** | P0（必须实现） |

**验收标准：**
- [ ] 支持 OpenAI 官方 API
- [ ] 支持 Moonshot (月之暗面) API
- [ ] 支持 DeepSeek API
- [ ] 支持本地 Ollama 服务
- [ ] 切换服务商只需修改 `.env`，无需改代码

---

#### FR-005 内置调参默认值

| 属性 | 说明 |
|------|------|
| **需求编号** | FR-005 |
| **需求描述** | 提供合理的 LLM 调用参数默认值，当前版本不通过环境变量暴露 |
| **内置参数** | `temperature`、`max_tokens`、`timeout`、`top_p` |
| **默认值** | 见下表 |
| **优先级** | P1（建议实现） |

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `temperature` | `0.7` | 平衡创造性和确定性 |
| `max_tokens` | `4096` | 单次响应最大长度 |
| `timeout` | `60` | 请求超时时间（秒） |
| `top_p` | `1.0` | 核采样，默认不限制 |

**验收标准：**
- [ ] 不配置环境变量时，使用上述默认值成功发起请求
- [ ] 默认值在代码中集中定义，便于统一调整

---

### 2.3 模板文件管理

#### FR-006 `.env.example` 模板维护

| 属性 | 说明 |
|------|------|
| **需求编号** | FR-006 |
| **需求描述** | 提供标准化的 `.env.example` 模板文件，包含所有必配项的占位符和注释 |
| **文件位置** | 项目根目录 |
| **提交仓库** | 是 |
| **优先级** | P0（必须实现） |

**验收标准：**
- [ ] 模板包含所有必配项（FR-001 定义）
- [ ] 每个变量附带中文注释说明用途和示例值
- [ ] 示例值使用明显无效的占位符（如 `your_api_key_here`）
- [ ] 按功能区块分组，格式统一

---

#### FR-007 `.gitignore` 配置

| 属性 | 说明 |
|------|------|
| **需求编号** | FR-007 |
| **需求描述** | `.env` 文件必须被 Git 忽略，防止敏感信息泄露 |
| **忽略规则** | `.env`、`.env.local`、`.env.*.local` |
| **优先级** | P0（必须实现） |

**验收标准：**
- [ ] `git status` 不显示 `.env` 为未跟踪文件
- [ ] `git check-ignore .env` 返回匹配结果

## 3. 非功能需求 (Non-Functional Requirements)

### 3.1 安全性

#### NFR-001 敏感信息保护

| 属性 | 说明 |
|------|------|
| **需求编号** | NFR-001 |
| **需求描述** | API Key 等敏感信息不得出现在代码、日志或错误信息中 |
| **适用范围** | 代码仓库、日志文件、错误堆栈、调试输出 |
| **优先级** | P0（必须实现） |

**验收标准：**
- [ ] 日志中打印配置时，API Key 显示为 `sk-***...***`
- [ ] 异常信息不包含完整的 API Key
- [ ] 代码审查时可通过简单搜索确认无硬编码密钥

---

#### NFR-002 配置隔离

| 属性 | 说明 |
|------|------|
| **需求编号** | NFR-002 |
| **需求描述** | 不同环境（开发/测试/生产）使用独立的配置，互不干扰 |
| **实现方式** | 支持 `.env.development`、`.env.production` 等覆盖文件 |
| **优先级** | P1（建议实现） |

## 5. 验收标准汇总

### 5.1 启动检查清单

- [ ] 复制 `.env.example` 为 `.env` 并填入有效参数后，应用正常启动
- [ ] 删除任一必配项后，应用启动失败并给出明确提示
- [ ] 切换不同服务商的 `LLM_BASE_URL` 和 `LLM_MODEL`，无需改代码即可运行

### 5.2 安全检查清单

- [ ] `.env` 不在 Git 跟踪范围内
- [ ] 日志和错误信息中不暴露完整 API Key
- [ ] `.env.example` 中不包含真实密钥
